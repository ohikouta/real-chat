# Roadmap

この文書は、`Real Chat` の次フェーズを整理するためのロードマップです。

## ゴールの考え方

このアプリは放置するとすぐに化石化します。次フェーズでは「何のために触り続けるのか」を先に固定します。

現時点で妥当なゴールは次の 3 段です。

### Phase 1. 現状整理

- 実装の評価
- 開発の振り返り
- ドキュメント整理
- 今後の課題の可視化

### Phase 2. 公開可能状態

- deploy 手順を固める
- 初回デプロイを行う
- 公開 URL で主要導線を確認する

### Phase 3. 継続改善

- UX 改善
- パフォーマンス改善
- テスト整備
- 運用しやすさの向上

## 次に切るべき Issue 候補

### 1. 全体評価と振り返りを docs に整理する

目的:

- 実装の現在地を文章で固定する
- よかった点、問題点、改善方針を残す

成果物:

- `docs/project-assessment.md`
- `docs/project-retrospective.md`
- README からの導線

### 2. デプロイ手順を整理して初回公開の前提を固める

目的:

- どこに、何を前提に、どう公開するかを定義する

成果物:

- `docs/deployment-plan.md`
- deploy 前チェックリスト
- 必要な Firebase 設定メモ

### 3. Firebase Hosting で初回デプロイする

目的:

- ローカルアプリを実 URL で触れる状態にする

成果物:

- Hosting 設定
- 公開 URL
- 初回デプロイ後の動作確認

### 4. 公開前の軽微な品質改善を行う

目的:

- 白画面や設定漏れのような初歩的な事故を減らす
- README とアセットの粗さを減らす

候補:

- env 設定の説明強化
- 画像や bundle size の見直し
- Console warning / error の棚卸し

### 5. 継続改善 backlog を整える

目的:

- 実装アイデアを放置せず、正式 Issue と保留メモに分ける

候補:

- UI / UX 改善
- テスト導入
- 監視 / 運用メモ
- Firebase Emulator を使った確認強化

## 優先順位

1. ドキュメントと振り返りの整理
2. デプロイ前提の整理
3. 初回デプロイ
4. 公開後の改善 backlog 化

## 進め方

- まず docs で現状を固定する
- その内容を GitHub Issue に落とす
- Issue 単位で worktree / branch を切って進める
- 公開後に、追加改善を第 2 フェーズとして継続する
